# CLAUDE.md

## 目的

このファイルは、AI支援ツールを使用する際に、このプロジェクトの前提を共有するためのガイドラインです。

このプロジェクトは、完成車メーカー再挑戦に向けて、C言語と車載組み込みの考え方を段階的に学ぶための学習用プロジェクトです。

AIは実装、整理、レビューの補助として使用しますが、目的はAIに作業を丸投げすることではありません。
最終的に、学習者本人がコードの流れ、責務分割、実装理由、確認結果を自分の言葉で説明できることを重視します。

---

## 基本方針

* 1回の作業で扱うテーマは小さくする
* C初心者が理解し、説明できる範囲で進める
* 実装量よりも、理解・確認・記録を優先する
* 既存の構成や責務分割を大きく崩さない
* コードとドキュメントの整合性を保つ
* 学習過程が第三者にも分かるように記録する

---

## 作業前に確認するもの

作業を始める前に、必要に応じて以下を確認します。

* README.md
* project_context.md
* study_plan.md
* learning_journal.md
* AI_workflow.md
* daily_log/ 配下の最新Day
* src/
* include/
* Makefile

特に、daily_log の最新Dayに書かれている「次回やること」や「次回の設計方針」は、次の作業を考える出発点として扱います。

ただし、前回の方針を機械的に実行するのではなく、現在のコード・ドキュメント・理解状況を確認したうえで、当日の作業内容を判断します。

---

## 作業の進め方

いきなり実装に入るのではなく、まず現在の状態を確認します。

作業前に学習者が以下を自分で確認・整理する。AIは内容を把握するにとどめ、代わりに整理はしない。

* 現在の実装状況
* 前回ログの方針
* 今日進めるべき作業
* 変更が必要になりそうなファイル
* 実装後に確認すること

コードやドキュメントにズレがある場合は、新機能追加よりも整合性確認や小さな修正を優先します。

直近の実装を説明できていない場合は、新しい実装よりも理解確認を優先します。

---

## 実装時の考え方

実装時は、各モジュールの責務を確認してから変更します。

既存モジュールの最新の責務は、README.md または project_context.md の内容を正とします。

基本方針は以下です。

* 1つのモジュールに複数の責務を詰め込みすぎない
* 新しい処理を追加する場合、既存モジュールに入れるべきか、新しい `.c` / `.h` に分けるべきかを検討する
* `main.c` には個別機能の詳細ロジックを増やしすぎず、初期化・処理順・ループ制御を中心にする
* センサ値、状態判定、統計、警告、ログ、診断、通信などの責務が混ざらないようにする
* 既存のデータの流れを壊さない
* 読むだけの引数には、可能な範囲で `const` を使う
* 新しい `.c` ファイルを追加した場合は、Makefile の更新漏れがないか確認する
* `gcc -Wall -Wextra -std=c11` で警告が出ない状態を維持する

---

## ドキュメント方針

コードを変更した場合は、README.md、project_context.md、daily_log などの内容と実装にズレがないか確認します。

ドキュメントは、AIに丸ごと代筆させるのではなく、学習者本人が理解した内容を自分の言葉で記録することを重視します。

AIは、以下のような補助に使います。

* コードとドキュメントにズレがないか確認する
* 更新が必要そうな箇所を指摘する
* 説明が不足している箇所を見つける
* 学習者が理解を整理するための観点を出す

各ファイルの役割は以下の通りです。

* `CLAUDE.md`：AI支援ツールに共有するプロジェクトの前提・作業ルール（このファイル）
* `AI_workflow.md`：学習者本人のAI活用方針
* `README.md`：第三者向けの概要、機能、構成、実行方法
* `project_context.md`：現在の実装状況、モジュール構成、次回作業の前提
* `study_plan.md`：長期的な学習計画とPhase構成
* `learning_journal.md`：学んだことを短く整理する理解メモ
* `daily_log/`：その日の作業内容、理解したこと、曖昧なこと、次回方針

README.md は概要資料として保ち、daily_log のように細かく書きすぎないようにします。
learning_journal.md は、後から見返すための短い理解メモとして整理します。

---

## AI_workflow.md との役割分担

`CLAUDE.md` は、AI支援ツールにプロジェクトの前提や作業上の注意点を共有するためのガイドラインです。

`AI_workflow.md` は、学習者本人がAIをどのような目的・方針で使うかを整理するための文書です。

役割は以下の通りです。

* `CLAUDE.md`：AIに共有する作業上の前提・注意点
* `AI_workflow.md`：自分のAI活用方針

## 確認・検証

実装後は、原則として以下を確認します。

```bash
make clean && make
```

必要に応じて、実行確認も行います。

```bash
make run
```

確認後は、学習者が以下を自分の言葉でまとめる。AIは確認と抜け漏れの指摘のみ行う。

* 変更したファイル
* 変更内容
* ビルド結果
* 実行結果
* 学習者が自分の言葉で説明できるか確認すべきポイント（AIが提示し、学習者が確認する）
* ドキュメントとの整合性

---

## 避けること

以下は避けます。

* 現在のPhaseや理解度を無視して、大きすぎる実装に進む
* C初心者が説明できない設計・実装にする
* 既存の責務分割を確認せずに機能を追加する
* 大規模リファクタリングを行う
* コードだけ変更して、README.md / project_context.md / daily_log とのズレを放置する
* daily_log に事実と違うことを書く
* README.md を日報のように長くする
* learning_journal.md を長文化する
* Makefile の更新を忘れる
* ビルド確認を省略する
* 見栄えのためだけに、理解できていない高度な実装を追加する
* AIが作っただけに見える不自然なコメントや説明を大量に入れる

高度なテーマを扱う場合も、現在のPhase、理解度、直近のdaily_logと整合しているかを確認してから進めます。

---

## 最重要方針

このプロジェクトでは、完成品の大きさよりも、学習過程と説明可能性を重視します。

* 小さく実装する
* 動かして確認する
* 何をしたか記録する
* なぜそうしたか説明できる
* 分からないことを分からないまま記録する
* 次回の判断につなげる

AIは作業を補助するために使います。
学習者本人の理解・判断・説明を置き換えるためには使いません。
