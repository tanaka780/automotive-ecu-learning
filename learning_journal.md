## C言語基礎

### struct

関連するデータをまとめる型。speed / rpm / temperature を1つにまとめることで、関数に渡すときに1つ渡すだけで済む。

### typedef

`struct VehicleSensorData` と毎回書かずに `VehicleSensorData` だけで使えるようにする仕組み。型に短い別名をつける。

### .h ファイル

他のファイルに関数を公開するための宣言を書く場所。実装（中身）は .c に書く。

### Makefile

ビルドを自動化する仕組み。`make` だけで複数ファイルのコンパイルをまとめて実行できる。

### & とポインタ

- `&sensor_data` は変数に自動で割り当てられた住所（番地）を調べる演算子
- `VehicleSensorData *data` は住所を受け取るための変数（ポインタ）。値ではなく住所そのものが入っている
- 値をそのまま渡すとデータのコピーが作られる。コピーを書き換えても元データは変わらない。住所を渡すと元データを直接書き換えられる

### -> 演算子

住所（ポインタ）の先にある箱の特定のメンバーにアクセスする。読み取りにも書き込みにも使う。
例: `data->speed` → data に入っている住所の先にある speed にアクセスする

### const ポインタ

`const VehicleSensorData *data` はポインタ経由での書き込みを禁止する。
「読むだけ」の責務を型で明示するために使う。

### 整数除算と型変換

- `int ÷ int` はコンパイラが演算時点で int として計算する（小数点以下は切り捨て）
- `float result = int_a / int_b` は int 除算の後に float に変換するため、結果は小数にならない
- float の精度が必要な場合は `(float)int_a / int_b` のようにキャストが必要

### enum（列挙型）

複数の状態を名前で表す型。数値（0, 1, 2）をそのまま使うと意味が伝わらないため、
`LEVEL_NORMAL` / `LEVEL_WARNING` / `LEVEL_CRITICAL` のように名前をつける。
typedef と組み合わせて `SensorLevel` という型名をつけることができる。

### static 関数

`static` をつけた関数は、そのファイルの中だけで使える内部専用関数になる。
外部に公開しない補助関数（ヘルパー関数）には static をつけるのが C 言語の慣習。
status.c の `classify_level` と `level_to_str` はこのパターンで実装した。

### 固定幅整数型

車載 C では `int` の代わりに `<stdint.h>` の `uint8_t` / `uint16_t` 等を使う慣習がある。
`int` はプラットフォームによってサイズが変わるが、`uint8_t` は常に 8 ビット符号なしと確定するため、
値の範囲と符号を型で明示することで意図しない型変換やオーバーフローを防ぐ。

#### 実装での型選定（Day5）

| フィールド | 値域 | 選定型 | 根拠 |
| --- | --- | --- | --- |
| `speed` | 0〜120 km/h | `uint8_t` | 最大 255 で十分 |
| `rpm` | 0〜6000 rpm | `uint16_t` | 6000 > 255 なので 8 ビットでは足りない |
| `temperature` | 25〜100 ℃ | `uint8_t` | 最大 255 で十分 |
| `speed_sum` | 最大 20×120=2,400 | `uint16_t` | 最大 65,535 で十分 |
| `rpm_sum` | 最大 20×6000=120,000 | `uint32_t` | 120,000 > 65,535 なので 16 ビットでは足りない |
| `temp_sum` | 最大 20×100=2,000 | `uint16_t` | 最大 65,535 で十分 |

**min 初期値の変更**:
元の `INT_MAX` は `int` 向けの最大値だった。`uint8_t` 型には `UINT8_MAX`（255）、
`uint16_t` 型には `UINT16_MAX`（65535）を使う。これらは `<stdint.h>` で定義されている。

**printf でのキャスト**:
`uint32_t` を `%d` に渡すと未定義動作の可能性があるため `(int)` キャストを明示する。
`uint8_t` / `uint16_t` は可変長引数渡し時に自動昇格されるが、意図を明示するため同様にキャストした。

## 設計パターン

### 責務分割

1つのモジュール（.c ファイル）が持つ仕事を1つに限定する。

- `sensor.c`：今この瞬間のセンサ値を管理する
- `stats.c`：これまでの値の最小・最大・合計を管理する
- `alert.c`：閾値チェックと警告出力のみを担当する
- `status.c`：センサ値の状態分類（NORMAL / WARNING / CRITICAL）と表示を担当する

混ぜると「このファイルは何をするファイルか」を一言で説明できなくなる。
また、一方を修正したときにもう一方に影響するリスクが生まれる。

### alert.c と status.c を分けた理由

一見、どちらも「閾値を超えたら何かする」に見えるが、責務が異なる。

- `alert.c`：閾値を超えた瞬間に警告を出力する（即時出力、2値判定）
- `status.c`：センサ値を3段階に分類し、毎周期の状態として表示する（常時表示、状態管理）

`SensorStatus` struct はセンサの生の値は持たず、分類結果（SensorLevel）だけを持つ設計にした。
こうすることで status.c の責務が「分類と表示」だけに絞られ、センサ値を直接触ることがない。

### 状態遷移の検出（エッジ検出）

「CRITICALである」ことと「CRITICALになった瞬間」は別の情報。後者を検出するには、前回の状態と今回の状態を比較する必要がある。

最初はこの比較用データ（previous/current）をstatus.cに持たせたが、status.cの責務は「センサ値を渡されたらその場で分類するだけ」だったため矛盾すると気づき、元に戻した。前回の状態は、それを使う側（diag.c）が自分の中で保持する設計にした。

### 並行変数（parallel variables）

`speed_crit_count` / `rpm_crit_count` / `temp_crit_count` のように、同じ意味の変数をセンサ別に名前だけで区別する書き方。

センサが増えるたびに変数と処理が増え、同じロジックを手書きで繰り返すことになる。enum + struct + 配列にまとめてループで処理する設計に直した。

### #define マクロ

定数に名前をつける仕組み。コンパイル前にプリプロセッサが数値に置き換える。
`ALERT_SPEED_MAX` のように書くことで、数値の意味がコードから読み取れる。
閾値を変更したいときは .h の定義箇所だけを修正すればよい。

## 車載ソフトウェアの考え方

### センサ値の物理的な範囲

センサ値の範囲は「実際の車がどこまで動くか」から決める。

- speed: 0〜120 km/h は停車〜一般道の上限
- rpm: 800〜6000 はアイドリング（エンジンがかかっている最低回転）〜レッドライン（上限）
- temperature: 25〜100℃ は常温〜冷却水が沸騰する手前

範囲が決まると「この型で足りるか」を計算できる。

### 実際のECUとシミュレーションの違い

実際の ECU はセンサや CAN という車内の通信線から値を読み取る。
このプロジェクトの `sensor_update()` はその部分を `rand()` のランダム値で代わりにやっている。

### 複数アラートの同時発火

`alert_check` の `if` を `else if` にすると、speed が引っかかった時点で rpm と temperature のチェックが飛ばされる。
`if` を3つ並べることで、速度・回転数・水温が同時に閾値を超えてもそれぞれ警告を出せる。
実際の車でも複数の異常が同時に起きることはある。

### 段階的なアラート設計

alert.c の単一閾値は「超えた / 超えていない」の2択でシンプルだが、緊急度の区別ができない。
status.c の2段階閾値にすると NORMAL / WARNING / CRITICAL を区別でき、注意段階で早めに気づける。
実際の ECU でも DTC（故障診断コード）は重篤度が複数段階に分かれている。

### DTC（故障診断コード）の基本的な考え方

DTCは「閾値を超えた」という事実そのものではなく、「異常が新たに発生した」というイベントを記録するもの。CRITICALが続く間ずっと記録すると、本来1件の故障が何十件にも増えてしまう。

前回の状態と今回の状態を比較し、NORMAL/WARNINGからCRITICALに変わった瞬間だけを記録することで、実際のECUの故障診断に近い動作になる。

### 固定幅整数型の車載での意義

`int` は環境によってサイズが変わるため、プラットフォームが違うことが多い車において、`int` だと想定外の動きをするリスクがある。
`uint8_t` や `uint16_t` を使えば「何ビットか」「符号ありかなしか」が型の名前から分かる。
